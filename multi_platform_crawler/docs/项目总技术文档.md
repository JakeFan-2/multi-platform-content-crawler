## 1. 项目概述

### 1.1 项目定位
多平台内容数据采集系统，面向内容运营人员，支持从 **11 个稳定平台**（微信公众号、微博、头条号、知乎、百家号、网易号、企鹅号、一点资讯、ZAKER、雪球、极客公园官网）自动采集指定文章的**阅读量、点赞量、评论量、转发量、曝光量**等核心运营数据。

### 1.2 核心设计原则
| 原则 | 说明 |
|------|------|
| **无侵入扩展** | 不修改原有采集器核心代码，仅通过标准接口调用 |
| **配置驱动** | 所有选择器、URL、超时、重试均在 YAML 中定义，代码零修改应对页面改版 |
| **线程安全** | Qt 主线程 + QThread 子线程 + asyncio，跨线程仅通过信号/槽通信 |
| **安全合规** | Cookie AES 加密存储，敏感信息环境变量化，日志自动脱敏 |
| **故障隔离** | 单平台异常不影响其他平台，自动记录快照并跳过 |

> **登录逻辑说明（严格区分场景）**：
> - **GUI 自动采集模式**：微信公众号、企鹅号、雪球号仅支持 Cookie 复用，Cookie 失效时自动停止登录流程，由调度器加入「需要手动登录的平台」队列，等待用户触发手动登录采集

> - **其余平台**：两种模式下 Cookie 失效均支持自动填充账号密码登录

## 4. 整体执行流程

### 4.1 主流程图（文本序列）

```
[用户启动程序]
    │
    ▼
[选择平台(≥1) + 输入标题(≤5) ]  (若头条/微博且标题>30字，展开关键词槽)
    │
    ▼
[点击"启动采集"]
    │
    ▼ (主线程发射 start_signal)
    │
    ▼
[CrawlThread 接收信号，在子线程 asyncio 循环中执行]
    │
    ▼
[CrawlScheduler.run() 获取平台队列(PlatformRegistry)]
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│  for each platform in queue:                           │
│     1. 调用采集器 crawl() 传入 headless=True            │
│     2. 采集器内部调用 ensure_login(page, context, headless) │
│        ├─ Cookie有效 → 继续采集                         │
│        ├─ Cookie失效 + 微信公众号/企鹅号/雪球平台：        │
│        │   └─ headless=True → 返回False，触发login_failed_callback │
│        │   └─ headless=False → 执行perform_login扫码登录 │
│        └─ Cookie失效 + 其余平台：                        │
│            └─ 自动填充账号密码登录 → 失败则加入manual_queue │
│     3. login_failed_callback 触发：                     │
│        └─ 平台加入 manual_login_queue，发出 login_required（调度器）│
│           与 CrawlThread.login_required_signal（至 GUI）      │
│     4. 调度器跳过该平台，继续下一平台                    │
│     5. 调用采集器 crawl() 执行策略一                    │
│        ├─ 自上而下遍历文章列表 (约60篇范围)            │
│        ├─ 标题匹配成功 → 提取数据 → 回调 result_callback│
│        ├─ 匹配完所有目标 → 提前终止该平台               │
│        └─ 遍历结束仍有未匹配 → 记录未匹配列表           │
│     6. 数据归一化: 11字段映射 + 从配置文件读取注入静态曝光量   │
│     7. 发送 data_signal 更新 GUI 表格                  │
│     8. 平台结束，继续下一个                            │
└─────────────────────────────────────────────────────────┘
    │
    ▼
[所有平台处理完毕] (包括手动登录队列中的平台待用户触发)
    │
    ▼
[执行时间兜底逻辑：雪球号/微信公众号发布时间替换]
    │ 雪球号：按优先级 zhihu → weibo → toutiao 查找有效时间
    │ 微信公众号：按优先级 website → qiehao → baijia → wangyi 查找有效时间
    │ 无有效时间来源 → 置空字符串
    │
    ▼
[CrawlThread 发射 finish_signal；CrawlScheduler 发射 task_finished]
    │
    ▼
[主线程 task_finished() 写日志汇总；导入飞书按钮在当前实现中启动后即可用]
    │
    ▼
[用户可点击"导入飞书" → 启动 FeishuImportWorker 子线程]
    │
    ▼
[FeishuExporter 批量写入飞书多维表格，通过信号返回结果]
```

### 4.2 手动登录与「手动登录并采集」流程（与当前代码一致）

**路径 A — 自动任务中 Cookie 失效（无头）**

```
[execute_platform 中 login 失败] → login_failed_callback
    │
    ▼
[平台写入 manual_login_queue，发射 login_required / login_required_signal]
    │
    ▼
[主线程 add_manual_login_platform，可选弹出 ManualLoginDialog]
    │
    ▼
[用户在对话框确认完成] → on_manual_login_complete → 再 submit_task(execute_manual_crawl, …)
```

**路径 B — 区域4「手动登录采集」或 区域5「手动登录并采集」**

```
[用户已选平台，且主界面至少 1 条标题]
    │
    ▼
[主线程发射 manual_login_signal(platform_id)]
    │
    ▼
[main._on_manual_login_clicked → CrawlThread.submit_task(scheduler.execute_manual_crawl, …)]
    │
    ▼
[若整任务 run() 正在进行且未处于用户暂停：先 pause()，结束后再 resume()]
    │
    ▼
[execute_manual_crawl → execute_platform(..., headless=False)]
    ├─ load_collector 加载 exe 同级外置 collectors/<platform>.py（打包模型友好）
    ├─ 采集器 ensure_login + crawl，有头浏览器下完成扫码/填表（与采集器实现一致）
    └─ 单条数据经 result_callback → thread.emit_data → 主线程表格
```

> 说明：`CrawlScheduler.manual_login()`（async、挂起等待 login_success_signal）用于**对话框链路**等场景；**区域4/5 按钮**不调用该方法，直接进入 `execute_manual_crawl`。各采集器文件内 `if __name__ == "__main__"` 的独立调试入口（如 DEBUG 列表写 CSV）与 GUI 主流程无关。

### 4.3 策略一详细匹配逻辑

```
输入: target_titles = ["文章A", "文章B", ...] (最多5个)
      keywords_map = {"文章A": ["关键词1","关键词2"]} (仅头条/微博超30字时)
      article_limit = 约60篇 (由各平台配置的翻页/滚动范围决定)

待匹配列表 = copy(target_titles)
结果列表 = []
遍历 article_list (自上而下):
    for 当前文章 in 文章列表:
        title = 提取标题
        matched_title = None
        for target in 待匹配列表:
            if 使用关键词匹配模式:
                if any(keyword in title for keyword in keywords_map[target]):
                    matched_title = target
                    break
            else:
                if 三级匹配(title, target):  # 精确/清理/标准化
                    matched_title = target
                    break
        if matched_title:
            提取完整字段 (url, read, like, comment, ...)
            结果列表.append(归一化数据)
            待匹配列表.remove(matched_title)
            if not 待匹配列表:
                break  # 提前终止
        # 否则继续下一篇文章
    if 达到范围上限或翻页结束:
        break
返回: 结果列表, 待匹配列表 (未匹配的标题)
```

## 5. 核心模块详解

### 5.1 调度中心 (CrawlScheduler)

**文件**: `controller.py`  
**类**: `CrawlScheduler` (继承 QObject)  
**生命周期**: 在子线程 `CrawlThread` 中实例化，运行独立 asyncio 事件循环。

**核心属性**（`controller.py` 实际字段）:
```python
self.platform_queue = []          # 待采集平台标识列表 (FIFO)
self.manual_login_queue = []      # 自动登录失败的平台
self.current_platform = None      # 当前正在执行的平台
self.is_paused = False
self.is_running = False           # run() 是否在进行
self.all_data / self.all_unmatched # 汇总结果（run 开始时清空）
```

无头/有头由每次调用 `execute_platform(..., headless=...)` 传入；采集器侧可通过 `set_headless_override` 接收。**无**调度器级的 `self.headless_override` 布尔字段。

**关键方法**:

| 方法 | 功能 |
|------|------|
| `set_platforms(platforms)` | 设置待采集队列（经 PlatformRegistry 校验） |
| `run(targets_with_index, keywords)` | 主循环：逐平台 `execute_platform`（默认无头） |
| `execute_platform(platform, targets, headless, title_restore_map=…)` | 单平台：load_collector → ensure_login → crawl → 回调归一化写表 |
| `validate_targets(targets_with_index, platform)` | 头条/微博超 30 字关键词校验等 |
| `execute_manual_crawl(platform, targets_with_index)` | 手动有头采集；可暂停/恢复自动队列 |
| `normalize_data(...)` | 字段映射 + 曝光量注入（雪球标题还原、ZAKER 前缀处理等） |
| `manual_login(platform)` | async：发登录请求并等待 `login_success_signal`（对话框链路） |
| `on_manual_login_complete(platform)` | GUI 对话框完成时解除等待并 `emit_login_success` |
| `_apply_time_fallback` / `_is_valid_time` / `_normalize_title` | 时间与标题兜底 |
| `pause()` / `resume()` / `stop()` | 队列控制；`stop()` 清空 `platform_queue` |

**CrawlScheduler 信号**（日志走 `CrawlThread.log_signal`，不在此列）:
```python
platform_started = Signal(str)
platform_finished = Signal(str, bool)
login_required = Signal(str)
login_success = Signal(str)
data_ready = Signal(dict)
unmatched_ready = Signal(str)
progress_updated = Signal(int, int, str)
error_occurred = Signal(str)
task_finished = Signal(list, list)   # (all_data, all_unmatched)
```

```python
class FeishuExporter:
    async def export_data(self, data: List[Dict]) -> Dict:

        # 2. 转换每条记录为中文字段名

```

### 5.4 安全模块 (utils/security.py)

**文件**: `utils/security.py`  
**类**: `SecureStorage` (单例模式)  
**核心功能**: 敏感数据加密存储与解密

**.encryption.key 密钥文件**:
- 自动生成于项目根目录
- 使用 Fernet 对称加密算法（AES-128 + HMAC）
- 密钥加载优先级：
  1. 环境变量 `ENCRYPTION_KEY`（base64 编码）
  2. `.encryption.key` 文件
  3. 均不存在时自动生成并保存

**核心方法**:
| 方法 | 功能 |
|------|------|
| `encrypt(data)` | 加密字符串，返回 base64 编码 |
| `decrypt(encrypted_data)` | 解密字符串 |
| `encrypt_dict(data)` | 加密字典（JSON 序列化后加密） |
| `decrypt_dict(encrypted_data)` | 解密字典 |

**应用场景**:
- Cookie 加密存储（`cookies/*.enc` 文件）
- 登录凭证加密
- 日志脱敏处理

## 7. 线程安全与跨线程通信

### 7.1 架构图

```
┌──────────────────┐         信号/槽 (自动排队)         ┌──────────────────┐
│   Qt 主线程      │ ◄───────────────────────────────► │  采集子线程       │
│  (GUI 事件循环)  │                                   │ (asyncio 循环)    │
│                  │                                   │                  │
│  - 接收用户点击   │                                   │ - 调度中心        │
│  - 更新界面控件   │                                   │ - 调用采集器      │
│  - 弹窗提示       │                                   │ - 网络请求        │
└──────────────────┘                                   └──────────────────┘
```

### 7.2 通信规则

| 方向 | 方式 | 示例 |
|------|------|------|
| 主线程 → 子线程 | 发射信号，子线程槽函数执行 | `self.start_signal.emit(platforms, titles, keywords)` |
| 子线程 → 主线程 | 发射信号，主线程槽函数更新界面 | `self.log_signal.emit("采集完成")`（当前为单字符串） |
| **禁止** | 子线程直接操作 Qt 控件 | ❌ `self.text_browser.append("xxx")` |
| **禁止** | 主线程阻塞等待子线程 | ❌ `self.thread.wait()` |

### 7.3 线程安全的数据存储

- **文件写入**: 使用 `threading.Lock` 保护 `CSVExporter` 写入操作。
- **Cookie 加解密**: 无状态，无竞态条件。
- **日志**: loguru 自身线程安全，直接调用。

**外置资源路径**：`collectors/`、`platforms/`、`config/`、`.env` 等均相对 **exe 所在目录**（`path_helper.PROJECT_ROOT`），与《交付、打包、迁移…》文档一致。

### 10.3 平台界面改版应对方案（最核心）

**现象**: 登录失败、找不到文章列表、提取不到数据。  
**原因**: 目标平台修改了 HTML 结构，导致 YAML 中的选择器失效。

**应对流程（运维人员操作，无需开发）**:

1. **定位失效平台**: 查看日志或 GUI 日志窗口，确定哪个平台失败。
2. **打开对应配置文件**: 例如 `platforms/toutiao.yaml`。
3. **只修改选择器部分**（不动其他任何配置，微信公众号/企鹅号/雪球平台登录逻辑不做修改）:
   ```yaml
   # 原来是:
   login:
     username_selectors:
       - css: "input[name='username']"
   # 改为新页面的选择器:
   login:
     username_selectors:
       - css: "input[placeholder='手机号/邮箱']"
       - id: "login-account"
   ```
4. **保存文件**，重启程序，重新采集。

**无需**: 修改代码、重新打包、安装任何环境。  
**故障隔离**: 一个平台改版只影响该平台，其他平台正常运行。

### 10.4 选择器快速定位技巧（给运维的指南）

- **Chrome 开发者工具**: F12 → 点击元素 → 右键 Copy → Copy selector / Copy XPath
- **优先使用 id**: `#username` 最稳定
- **次选 css**: `input[type='text']` 或 `.login-input`
- **兜底 role**: `role="textbox" name="账号"` (语义化)
- **测试选择器**: 在浏览器 Console 执行 `document.querySelector('css选择器')` 验证

### 10.5 常见故障速查表

| 故障现象 | 可能原因 | 解决方法 |
|----------|----------|----------|
| 启动后白屏 / 提示浏览器可执行文件不存在 | 未按 §8.3 部署 `%LOCALAPPDATA%\ms-playwright`，或与打包 Playwright 版本不一致 | 按 `首次配置说明.txt` 复制交付包内 `ms-playwright`；查 `logs/bootstrap.txt` 中 `PLAYWRIGHT_BROWSERS_PATH` |
| 提示“登录失败” | 通用：Cookie 失效；其余平台：自动登录选择器过时 | 微信公众号/企鹅号/雪球平台：点击手动登录完成扫码；其余平台：更新 YAML 中的 login 选择器，或手动登录 |
| 采集不到文章 | 文章列表选择器失效 | 更新 article_list.row_selectors |
| 某个字段为空 | 字段选择器失效 | 更新 field_selectors 中对应项 |
| 曝光量数值不符合预期 | 配置文件数值未更新 | 修改 `config/exposure.yaml` 中对应平台的数值，重启程序即可 |
| 程序崩溃 | 异常快照已保存 | 查看 snapshots/ 下截图和日志，联系开发 |
