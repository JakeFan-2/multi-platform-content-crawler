        │  - log_signal(str)  → 追加到 QTextBrowser    │

**关键设计说明**  
- 所有控件操作仅发生在主线程，子线程通过信号向主线程传递数据。  
- 区域 4 和区域 5 共用同一个 `manual_login_signal`，最终由 `main._on_manual_login_clicked` 统一处理，将任务投递到 `CrawlScheduler.execute_manual_crawl`。  
- 关键词输入框是动态生成的：当用户勾选微博或头条，且某一标题超过 30 个字时，才会在该标题行显示对应的关键词输入框。  
- 「导入飞书」按钮通过创建独立的 `FeishuImportWorker` 子线程执行，不会阻塞主界面。

### 5. 打包版用户操作流程图（完整业务流程）
面向最终用户的傻瓜式操作步骤，包括首次部署和日常采集。

```
[首次使用] 
   ├─ 按“首次配置说明”复制 ms-playwright 到 %LOCALAPPDATA%
   ├─ 修改 .env 中的账号、飞书等配置（可选）
   └─ 双击 exe 启动程序
[日常采集]
   1. 在区域1 勾选要采集的平台（至少1个）
   2. 在区域2 输入标题（不超过5个）
   3. 点击区域3 [启动采集]
      ├─ 自动任务运行（无头浏览器）
      ├─ 区域6 实时显示日志
      ├─ 区域7 逐步出现数据
      └─ 若某平台Cookie失效 → 平台名称出现在区域4
   4. 手动登录采集（当区域4有平台时）：
      a) 在区域4 选中平台 → 点击[手动登录采集]
         → 弹出有头浏览器，完成扫码/填表登录后自动采集
      b) 或直接在区域5 选中目标平台 → 点击[手动登录并采集]
         → 同样弹出有头浏览器完成登录及采集
   5. 全部完成后，点击区域8 [导出CSV] 或 [导入飞书]
   6. 关闭程序，下次启动自动复用登录态
```

### 6. 修正后的核心执行流程图（补充手动登录双路径）
该图完整覆盖了 `execute_platform` 中的登录分支，并显式加入了区域4和区域5触发的 `execute_manual_crawl` 路径。

```mermaid
graph TD
    A[用户点击启动采集] --> B[CrawlThread 启动调度]
    B --> C{遍历平台队列}
    C --> D[execute_platform (headless=True)]
    D --> E{登录状态}
    
    E -- Cookie 有效 --> F[执行策略一采集]
    E -- Cookie 失效 --> G{平台类型?}
    G -- 微信/企鹅/雪球 --> H[直接进入登录失败回调]
    G -- 其他平台 --> I[尝试自动填表登录]
    I -- 失败 --> H
    
    H --> J[加入手动登录队列 (区域4显示)]
    J --> K[发射 login_required_signal]
    K --> L{用户操作}
    
    L -- 区域4: 选中并点击“手动登录采集” --> M[主线程调用 execute_manual_crawl]
    L -- 区域5: 直接点击“手动登录并采集” --> M
    
    M --> N[execute_platform (headless=False)]
    N --> O[弹出有头浏览器，完成扫码/填表]
    O --> F
    
    F --> P[数据归一化 & 更新表格]
    P --> Q{还有未处理平台?}
    Q -- 是 --> C
    Q -- 否 --> R[执行时间兜底逻辑]
    R --> S[完成，发射 finish_signal]
```
